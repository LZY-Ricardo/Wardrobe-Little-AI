# Change: 为统一 Agent 增加图片直发对话 V1

## Why
当前 Agent 聊天已经支持文本多轮对话、历史会话、快捷入口和图片选取跳转到 `Add` 页，但用户仍然无法在聊天窗口内直接发送图片给 Agent。对移动端用户来说，“打开相册 -> 跳转到另一个页面 -> 再分析/再返回”的链路过长，也不符合主流多模态聊天产品的预期。

为了让 Agent 更像一个真正的智能穿搭助手，需要提供 V1 版本的“图片直发对话”能力：用户可以在聊天页直接选择图片，并结合文字一起发送给 Agent；Agent 在同一会话内理解图片并返回分析结果、穿搭建议或页面指导，而不是强制跳转到独立上传页面。

同时，当前项目里的图片消息处理仍偏向“服务端先固定做图片分析，再将摘要拼接回文本输入”的实现思路。这种做法虽然能快速复用已有能力，但会带来两个问题：
- 图片摘要中的关键词容易干扰统一 Agent 的任务分类，导致图文问答被错误路由到画像/推荐等任务分支。
- 图片消息即使进入 `chat-stream`，也可能因为前置分析和任务直返而失去真正的流式文本输出体验。

因此本次变更除了补齐图片直发聊天，还要升级为更标准的 Agent 形态：保留 `DeepSeek` 作为主对话模型，复用现有 `SiliconFlow Vision` 视觉模型作为图片解析工具，并让 LLM 在会话内自主决定是否调用该工具；服务端只提供少量兜底策略与工具编排能力。

## What Changes
- 新增 Agent 聊天页的图片待发送能力，支持从相册选图并在输入区展示预览。
- 新增统一 Agent 图文消息发送协议，支持“纯图”与“图 + 文本”两种输入。
- 新增服务端图文聊天处理链路，复用 `SiliconFlow Vision` 作为图片解析工具，支持 `DeepSeek` 在回复过程中自主决定是否调用图片分析工具。
- 新增统一 Agent 的工具调用编排层、图片工具结果回注机制与阶段流式事件。
- 调整带图消息的业务路由逻辑：普通图文问答默认走工具增强聊天链路，仅在用户文本明确表达业务任务时进入现有任务分支。
- 新增聊天消息中的图片消息展示与失败态、历史恢复兼容。
- 新增历史会话摘要对图片消息的兼容展示，避免出现不可读的原始对象或长 base64 预览。

## Impact
- Affected specs: `unified-agent`
- Affected code:
  - Frontend: `client/src/pages/AiChat`、输入区、消息卡片、更多操作菜单、历史预览摘要逻辑
  - Backend: `server/routes/unifiedAgent.js`、`server/controllers/unifiedAgentRuntime.js`、图片工具服务层、SSE 事件分发
  - Database: `agent_messages` 需要兼容图片类消息与图文混合消息的元数据
- Reuses existing capabilities:
  - unified-agent session / message persistence
- Reused external dependency:
  - `SiliconFlow Vision` 托管视觉服务（作为统一 Agent 图片工具）

## Scope
### Included in V1
- 单张图片发送
- 纯图发送
- 图片 + 文本一起发送
- 聊天页图片预览
- 发送后在会话中保留图片消息
- Agent 基于图片工具结果继续回复
- `DeepSeek` 可调用 `analyze_image` 工具，并在同一轮会话中继续生成最终回复
- 聊天 SSE 支持图片工具调用阶段事件与最终文本流

### Excluded from V1
- 多图发送
- 图片裁剪/编辑
- 真正的文件附件系统
- 聊天中直接上传到衣橱
- 图片消息的云端长期对象存储
- 图片消息撤回、转发、多选操作
- 通用工具平台的完整抽象框架
- 多工具并发调用或复杂工具链规划

## Compatibility
- 原有文本聊天能力保持不变。
- 当前“打开相册 -> Add 页”链路可保留为兼容入口，但加号菜单中的主入口应优先迁移到“图片直发聊天”。
- 原有任务 Agent 能力保持不变，但带图消息不再默认用图片摘要参与任务分类。

## Risks
- 图文消息会增加会话协议复杂度。
- 工具调用链路会增加流式编排复杂度。
- 图片工具失败时，需避免影响普通文本聊天稳定性。
- 若直接持久化 base64，消息表体积可能增长过快。
- 若模型滥用工具或重复调用，会引入额外成本与延迟。

## Mitigation
- V1 仅支持单图，限制体积和协议复杂度。
- 将图片工具限制为每条用户消息最多调用一次，并仅允许分析当前消息附件。
- 图片工具失败时降级为“图片理解失败，请结合文字继续描述”的明确回复，不影响会话本身。
- V1 优先存储结构化图片消息元数据与前端可恢复预览字段，不引入复杂附件平台。
- 为工具调用阶段增加显式 SSE 事件，避免前端表现为长时间无反馈。
