## ADDED Requirements
### Requirement: task-agent MUST 将工具确认策略收敛为独立策略模型
系统 MUST 将写操作确认、低风险免确认、删除类强制确认和批量写入确认等规则收敛为独立策略模型，而不是散落在运行时代码与执行桥接逻辑中。

#### Scenario: 删除类工具统一强制确认
- **WHEN** Agent 请求执行删除衣物、删除套装或删除穿搭记录等删除类工具
- **THEN** 系统通过统一策略模型判定该操作必须进入确认流，且不允许因用户偏好而跳过确认

#### Scenario: 低风险写工具由策略层决定是否免确认
- **WHEN** Agent 请求执行低风险写工具，例如收藏状态切换
- **THEN** 系统由统一策略模型判断是否允许免确认，而不是在多个控制器中重复硬编码判断

### Requirement: task-agent MUST 明确任务历史、确认流与兼容回退逻辑的职责归属
系统 MUST 将任务历史仓储、确认服务和兼容自然语言回退逻辑的职责边界显式定义，并避免继续将这些职责无边界堆叠在单一 Agent 控制器中。

#### Scenario: 待确认任务由独立确认服务恢复
- **WHEN** 用户恢复一个包含待确认写操作的会话
- **THEN** 系统通过独立确认服务与任务历史仓储恢复该任务，而不是依赖散落在控制器中的临时恢复逻辑

#### Scenario: 未进入 tool loop 的兼容任务由独立 fallback 服务处理
- **WHEN** 某个请求仍需要走兼容自然语言任务解析路径
- **THEN** 系统通过显式的 fallback 服务处理该逻辑，而不是继续将兼容分支与标准工具执行杂糅在同一控制器中

### Requirement: task-agent MUST 将上下文注入规则收敛为独立策略模型
系统 MUST 将“当前衣物 / 当前套装 / 当前穿搭记录 / 当前推荐结果 / 当前图片输入”等上下文注入规则收敛为独立策略模型，以便统一控制模型可见上下文与工具执行上下文。

#### Scenario: 工具调用前注入当前业务对象上下文
- **WHEN** 用户在会话中提到“这件 / 当前 / 刚刚那套”之类上下文对象
- **THEN** 系统通过统一上下文策略向 runtime 与 LLM 注入相应上下文，而不是在多个运行时分支中临时拼接

#### Scenario: 图片任务注入多模态上下文
- **WHEN** 当前会话携带图片附件且任务与图片内容有关
- **THEN** 系统通过统一上下文策略向 runtime 暴露图片工具和附件索引信息，而不是由调用方自行推断

### Requirement: task-agent MUST 将工具执行与业务实现按业务域拆分
系统 MUST 将工具执行逻辑按业务域拆分到独立 handler 文件，并复用现有业务层函数，而不是继续把实际执行逻辑集中堆叠在统一注册表或 Agent 控制器中。

#### Scenario: 衣物域工具由 wardrobe handler 执行
- **WHEN** Agent 调用衣物查询、衣物创建、衣物修改或批量保存衣物等工具
- **THEN** 系统通过 wardrobe 域的 handler 执行共享业务逻辑，而不是在统一注册表文件内直接堆叠全部实现

#### Scenario: 图片理解工具由 vision handler 执行
- **WHEN** Agent 调用图片分析工具
- **THEN** 系统通过 vision 域 handler 执行图片理解逻辑，并保持与衣物录入工作流兼容

### Requirement: task-agent MUST 使用统一的展示与确认描述模型
系统 MUST 使用统一的展示与确认描述模型生成工具执行摘要、确认卡片描述和过程状态文案，避免这些展示逻辑继续散落在 runtime、Agent 控制器或前端适配代码中。

#### Scenario: 工具结果摘要由 presenter 生成
- **WHEN** 某个工具执行完成并需要向前端或 LLM 回填结果摘要
- **THEN** 系统通过该工具对应的 presenter 生成标准摘要，而不是在多个调用点分别拼接摘要文本

#### Scenario: 确认卡片描述由 descriptor 生成
- **WHEN** 某个写工具进入确认流
- **THEN** 系统通过该工具对应的 confirmation descriptor 生成标题、摘要、范围和 details，而不是在多个控制器中手工拼装确认对象
