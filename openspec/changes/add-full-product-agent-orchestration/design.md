## Context
当前 unified-agent 已经完成了三个关键基础：

1. 统一会话、消息持久化、会话摘要与长期偏好摘要
2. 少量任务执行能力与写操作确认机制
3. 单张图片直发聊天与图片理解工具调用

但系统仍然保留“页面功能主导，Agent 局部增强”的结构。像衣物录入、衣物编辑、推荐保存、穿搭记录、画像调整、搭配预览等能力，依旧散落在各个页面与接口中，由用户自行串联。

本变更的目标不是继续增加几个任务关键词，而是把项目中的核心业务动作提升为统一 Agent 可调用的能力目录，并在 unified-agent 中提供受控的工作流编排能力。

## Goals / Non-Goals
- Goals:
  - 让 unified-agent 成为项目中所有核心手动操作的统一代理入口
  - 让 Agent 能基于文字、图片和当前上下文执行多步业务流程
  - 建立统一工具契约、风险分级和确认策略
  - 让页面功能与 Agent 能力共用同一套后端业务执行层
  - 保持执行过程可追踪、可解释、可恢复
- Non-Goals:
  - 不实现完全开放的自治型自由代理
  - 不允许模型直接绕过业务层访问数据库
  - 不在第一阶段覆盖项目中所有低价值或边缘操作
  - 不在本变更中引入多 Agent 协作

## Decisions
- Decision: 采用“统一 Agent Orchestrator + LLM 主导工具选择 + 标准业务工具 + 模板化工作流”的架构，而不是纯关键词路由或无护栏自由代理。
  - Why: 项目需要提升 Agent 的自主理解与跨功能编排能力，但同时保留明确的执行边界与可调试性。LLM 负责决定该调用哪些工具，runtime 负责校验、执行、确认和兜底。

- Decision: 页面能力与 Agent 能力共享同一业务执行层，Agent 不单独维护一套重复逻辑。
  - Why: 避免页面与 Agent 分叉演化，保持 DRY，并降低后续维护成本。

- Decision: 第一阶段优先覆盖高价值主流程，而不是一次性追求全站 100% 功能映射。
  - Why: 先把衣物录入、衣橱编辑、推荐保存、穿搭记录、画像调整等高频链路打通，收益最高。

- Decision: 继续保留确认机制，并扩展为统一风险分级模型。
  - Why: 当 Agent 代理全站操作后，误操作的代价更高，必须用统一安全边界约束。

## Architecture
### 1. Unified Agent Orchestrator
- 统一接收来自 unified-agent 的文本、图片、图文输入
- 负责工具目录注入、LLM 工具决策循环、参数校验、澄清判断、工作流调度和结果回写
- 统一协调普通对话、工具增强聊天、任务型工作流三种执行模式

### 1.1 LLM Tool-Calling Loop
- 由 LLM 在给定工具目录后自主决定是否调用工具、调用哪个工具、是否继续下一步
- runtime 负责：
  - 校验工具名是否在白名单中
  - 校验参数是否合法
  - 根据风险等级决定直接执行、进入确认流或拒绝执行
  - 将工具结果以结构化消息回填给模型，直到得到最终回复
- 第一阶段允许 LLM 自主调度只读工具和“待确认写操作意图”，但不允许绕过确认直接写库
- 当前实现已扩展为：真实接口下，所有登记工具都由 LLM 自主决定是否调用；写操作仍通过统一确认模型拦截

### 2. Tool Catalog
- 以“页面手动动作”为源头梳理标准工具
- 每个工具必须定义：
  - 名称
  - 输入 schema
  - 输出 schema
  - 所属业务域
  - 风险等级
  - 是否允许免确认
  - 是否可被多步工作流复用
  - 是否允许由 LLM 直接调用

### 3. Workflow Templates
- 对典型多步业务动作提供模板化工作流
- 第一阶段至少包含：
  - `ingest_cloth_from_image`
  - `update_cloth_from_text`
  - `recommend_and_save_suit`
  - `recommend_and_create_outfit_log`
  - `update_user_preferences`
  - `generate_match_preview`

### 4. Execution Guardrails
- 只允许通过标准工具执行业务读写
- 删除类操作始终确认
- 创建/修改类操作默认确认，少量低风险动作可配置免确认
- 工作流中的每一步都应记录中间状态，便于恢复和追踪

## Tool Domains
### 衣物域
- 读取衣橱列表、详情、筛选结果
- 识别图片并生成衣物草稿
- 创建衣物
- 更新衣物
- 切换收藏
- 删除衣物

### 推荐域
- 基于场景、天气、衣橱、偏好生成推荐
- 保存推荐为套装
- 将推荐落为穿搭记录
- 提交推荐反馈

### 套装域
- 查询套装
- 创建套装
- 更新套装
- 删除套装

### 穿搭记录域
- 查询穿搭记录
- 创建穿搭记录
- 更新穿搭记录
- 删除穿搭记录

### 画像与偏好域
- 获取画像
- 更新画像与确认偏好
- 读取衣橱分析与行为闭环摘要

### 预览域
- 根据用户模型图与衣物组合生成搭配预览

## Input Modes
### 纯文本
- 用于查询、推荐、编辑、帮助说明、偏好调整等

### 纯图片
- 用于衣物识别、基础理解、录入起点

### 图文输入
- 用于带约束的图片任务，例如：
  - “把这件衣服存入衣橱，风格设为通勤”
  - “这件衣服适合什么场景”
  - “按这件上衣生成搭配建议”

## Workflow Example
### 图片录入衣橱
1. 用户发送衣物图片
2. Agent 触发图片理解工具
3. Agent 生成结构化衣物草稿
4. 若关键字段缺失，进入澄清
5. Agent 返回待保存摘要
6. 用户确认
7. 调用创建衣物工具落库
8. 返回结果并写入任务历史

### 基于推荐直接落业务对象
1. 用户请求场景推荐
2. Agent 调用推荐工具
3. Agent 返回推荐结果
4. 用户继续要求“保存为套装”或“记录为今天穿搭”
5. Agent 进入确认流
6. 调用写工具执行并回写结果

## Risks / Trade-offs
- LLM 自主调用工具后，错误工具选择或错误参数会增加
  - Mitigation: 保留工具白名单、schema 校验、最大调用轮数、失败兜底回复和写操作确认护栏

- 工具数快速增加会提高编排复杂度
  - Mitigation: 先以业务域分组，采用显式工具目录与模板化 workflow，避免自由扩散

- 页面和 Agent 可能出现参数语义不一致
  - Mitigation: 把页面逻辑下沉为共享业务函数，避免双实现

- 图片录入工作流天然涉及识别不确定性
  - Mitigation: 允许“草稿 + 澄清 + 确认”三段式，而不是识别后直接写库

- 全站代理会放大误操作风险
  - Mitigation: 使用统一确认等级、不可绕过的删除确认、可追踪任务历史

## Migration Plan
1. 建立 Agent 工具总表和风险分级
2. 将真实接口主路径切换到统一 autonomous tool loop
3. 把图片录入、推荐保存、穿搭记录、画像与分析等高价值工具全部纳入 LLM 决策目录
4. 将旧关键词分流降级为兼容层
5. 让页面调用逐步复用相同业务执行层

## Open Questions
1. 第一阶段是否立即覆盖“删除衣物/删除套装/删除记录”？建议纳入工具目录，但默认保守上线
2. `genPreview` 是否作为第一阶段 Agent 可执行能力？建议纳入第二优先级
3. 会话内是否展示完整 workflow step 列表？建议第一阶段仅展示简化步骤摘要与关键状态
