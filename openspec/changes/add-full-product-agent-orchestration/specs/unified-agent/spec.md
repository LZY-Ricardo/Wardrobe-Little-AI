## ADDED Requirements
### Requirement: 统一 Agent MUST 成为项目级统一操作入口
系统 MUST 允许用户通过 unified-agent 访问项目中的核心业务能力，使衣橱、套装、推荐、穿搭记录、画像与预览等操作都能在统一会话中被代理执行。

#### Scenario: 用户在统一会话中执行页面级操作
- **WHEN** 用户在 unified-agent 中提出原本需要进入具体页面才能完成的业务请求
- **THEN** 系统在当前会话中完成该请求对应的业务编排，而不是强制要求用户跳转页面手动操作

#### Scenario: 页面快捷入口带上下文进入统一会话
- **WHEN** 用户从衣橱页、推荐页或画像页进入 unified-agent
- **THEN** 系统能够携带当前页面上下文和当前对象，减少用户重复描述

### Requirement: 统一 Agent MUST 支持工作流式业务编排
系统 MUST 在统一会话中支持多步业务工作流，包括任务识别、参数抽取、缺参澄清、工具调用、确认执行和结果回写。

#### Scenario: 图文输入触发录入工作流
- **WHEN** 用户发送衣物图片并表达“帮我存进衣橱”
- **THEN** 系统在当前会话中完成图片理解、草稿生成、必要澄清、确认和保存等连续步骤

#### Scenario: 文本输入触发推荐后续写入工作流
- **WHEN** 用户要求“推荐一套通勤穿搭并保存成套装”
- **THEN** 系统在当前会话中先生成推荐，再在确认后继续执行保存套装的写入步骤

### Requirement: 统一 Agent MUST 维护项目级工具目录与能力边界
系统 MUST 为 unified-agent 维护显式的工具目录、输入输出 schema、业务域归属和风险等级，以支持可控的任务路由与工作流复用。

#### Scenario: Agent 只调用已登记的业务工具
- **WHEN** unified-agent 判断需要执行某项业务能力
- **THEN** 系统只允许调用已登记在工具目录中的标准工具，而不是临时拼接或绕过业务层的内部逻辑

#### Scenario: 页面功能与 Agent 共享同一业务执行层
- **WHEN** 页面能力和 Agent 能力都需要执行同一业务动作
- **THEN** 系统复用同一后端业务执行层，保证行为和校验一致

### Requirement: 统一 Agent MUST 支持基于风险分级的执行控制
系统 MUST 按查询、推荐、创建、修改、删除等风险等级约束 unified-agent 的执行边界，并在统一会话中呈现相应的确认或执行状态。

#### Scenario: 查询与推荐可以直接执行
- **WHEN** 用户请求查询衣橱、查看画像或生成场景推荐
- **THEN** 系统可直接执行并返回结果，而无需额外确认

#### Scenario: 创建与修改按策略进入确认
- **WHEN** 用户请求创建衣物、创建套装、记录穿搭或修改偏好
- **THEN** 系统依据风险等级与用户偏好决定是否进入确认流，但必须保证执行前边界清晰

### Requirement: 统一 Agent MUST 支持工作流状态恢复与继续执行
系统 MUST 允许 unified-agent 在会话恢复后识别最近一次未完成或待确认的工作流状态，并支持继续澄清、继续确认或重新执行。

#### Scenario: 用户恢复待确认会话后继续执行
- **WHEN** 用户重新打开一段包含待确认写操作的 unified-agent 会话
- **THEN** 系统恢复该工作流状态，并允许用户继续确认或取消，而不是把状态丢失为普通聊天消息

#### Scenario: 用户恢复失败工作流后继续补充信息
- **WHEN** 某个工作流因缺参或图片识别不充分而未完成
- **THEN** 系统在恢复后保留该任务上下文，允许用户继续补充信息并推进下一步
