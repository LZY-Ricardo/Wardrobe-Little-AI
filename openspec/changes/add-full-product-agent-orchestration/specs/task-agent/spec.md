## MODIFIED Requirements
### Requirement: 系统 MUST 提供任务型 Agent 统一入口
系统 MUST 提供统一 Agent 页面、全局入口和业务页面快捷入口，使用户能够在不同页面上下文中通过文字、图片或图文组合发起任务，并逐步替代项目中的手动页面操作。

#### Scenario: 用户以文字发起跨页面业务任务
- **WHEN** 用户在统一 Agent 页面中输入“帮我把第 1 套推荐保存为套装”
- **THEN** 系统在当前会话中识别对应业务目标，而不是要求用户手动跳转到目标页面执行

#### Scenario: 用户以图片发起衣物录入任务
- **WHEN** 用户在统一 Agent 页面发送一张衣物图片并表达“存入我的衣橱”
- **THEN** 系统进入衣物识别与保存工作流，而不是要求用户手动进入 `Add` 页面完成上传和保存

### Requirement: 系统 MUST 支持基于上下文的任务解析与有限步骤规划
系统 MUST 结合用户上下文、当前页面对象、图片输入、长期偏好和最近会话状态，对用户请求进行任务解析，并在需要时生成有限步骤的工作流计划。

#### Scenario: Agent 将图文输入解析为多步录入工作流
- **WHEN** 用户发送衣物图片并附带“按通勤风格存进衣橱”
- **THEN** 系统将任务拆解为图片识别、草稿补全、必要澄清、确认保存和执行写入等步骤

#### Scenario: Agent 将组合写操作解析为串行工作流
- **WHEN** 用户要求“先推荐一套通勤穿搭，再帮我记成今天穿搭”
- **THEN** 系统将任务拆解为推荐生成、结果确认和创建穿搭记录等连续步骤

### Requirement: 系统 MUST 通过标准业务工具执行任务
系统 MUST 将项目中的核心手动操作沉淀为标准业务工具，并通过这些工具完成 Agent 任务，而不是直接绕过业务层访问数据库。

#### Scenario: Agent 通过衣物录入工具替代手动上传流程
- **WHEN** 用户要求将一张衣物图片保存到衣橱
- **THEN** 系统通过图片识别、衣物草稿生成和创建衣物等标准工具完成录入流程

#### Scenario: Agent 通过共享业务工具执行页面已有能力
- **WHEN** 用户要求修改偏好、创建套装或记录穿搭
- **THEN** 系统复用页面对应的后端业务能力完成执行，而不是维护与页面分叉的一套重复逻辑

### Requirement: 系统 MUST 对所有写操作执行统一确认控制
系统 MUST 对新增、修改、删除等写操作执行统一确认控制，并基于风险等级决定是否允许低风险动作进入免确认模式。

#### Scenario: 衣物录入在最终写入前需要确认
- **WHEN** Agent 已根据图片识别出衣物草稿并准备写入衣橱
- **THEN** 系统先返回待保存摘要，等待用户确认后再真正创建衣物记录

#### Scenario: 删除类操作始终不可免确认
- **WHEN** 用户要求删除衣物、删除套装或删除穿搭记录
- **THEN** 系统必须进入确认流程，且不允许因用户偏好配置而跳过确认

### Requirement: 系统 MUST 提供 Agent 任务历史与执行状态追踪
系统 MUST 记录用户最近的 Agent 任务摘要、步骤状态、确认状态和关联业务对象，供前端展示、恢复与错误排查。

#### Scenario: 多步工作流写入步骤状态
- **WHEN** Agent 执行“图片识别 -> 草稿确认 -> 保存衣物”这类多步任务
- **THEN** 系统记录至少任务摘要、当前步骤、确认状态和最终结果

#### Scenario: 工作流失败后保留可追踪状态
- **WHEN** Agent 在某一步骤中失败，例如图片识别失败或保存失败
- **THEN** 系统保留失败状态与结果摘要，供用户重试、补充信息或继续澄清
